## 시나리오 3: 데이터베이스 스키마 설계

**안소영 (백엔드):**

**류민지 (백엔드 시니어):**
> "EAV는 쿼리가 복잡해지고 성능도 떨어집니다. 차라리 JSONB 컬럼으로 스펙을 저장하는 게 낫지 않을까요? PostgreSQL의 JSONB 인덱싱 기능이 좋아졌으니까요."

**송재훈 (백엔드 시니어):**
> "동의합니다. 공통 필드(이름, 가격, 카테고리)는 정규 컬럼으로, 가변적인 스펙 필드는 JSONB로 저장하면 유연성과 성능을 둘 다 잡을 수 있습니다."

**황정민 (백엔드 리드):**
> "그럼 specs JSONB 컬럼에 스펙을 저장하고, 카테고리별 스펙 스키마를 별도 테이블로 관리해서 유효성 검증에 활용하겠습니다."

### 스키마 설계 결과

```sql
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    name_ko VARCHAR(100),
    slug VARCHAR(100) UNIQUE NOT NULL,
    parent_id INTEGER REFERENCES categories(id),
    spec_schema JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    name_ko VARCHAR(200),
    slug VARCHAR(200) UNIQUE NOT NULL,
    category_id INTEGER REFERENCES categories(id),
    description TEXT,
    specs JSONB,
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE product_images (
    id SERIAL PRIMARY KEY,
    product_id INTEGER REFERENCES products(id),
    url VARCHAR(500) NOT NULL,
    alt_text VARCHAR(200),
    is_primary BOOLEAN DEFAULT FALSE,
    sort_order INTEGER DEFAULT 0
);
```

# 갈등 해결 사례

## 사례 1: 일정 vs 품질 갈등

**상황:** 기획 변경으로 일정이 촉박해짐

**PM 박서연:**
> "기능 축소 없이 일정 준수는 어렵습니다."

**QA 리드 김은정:**
> "테스트 기간 축소는 품질 저하로 이어집니다."

**해결 과정:**
1. 전체 팀 긴급 미팅 소집
2. 기능별 우선순위 재정의
3. MVP(Minimum Viable Product) 범위 확정
4. Nice-to-have 기능 2차로 이관
5. 테스트 기간 유지, 대신 자동화 테스트 집중

**결과:** 핵심 기능만 1차 출시, 품질 유지, 일정 준수

## 사례 2: 디자인-개발 간 소통 문제

**상황:** 디자인 시안과 실제 구현물 차이 반복 발생

**UI 리드 강민수:**
> "개발팀에서 임의로 디자인을 변경합니다."

**프론트엔드 리드 오승우:**
> "실제 구현 시 기술적 제약이 있어서 조정이 필요합니다."

**해결 과정:**
1. 디자인 핸드오프 프로세스 정립
2. 디자인 QA 체크리스트 도입
3. 개발 착수 전 기술 검토 미팅 필수화
4. 변경 필요 시 반드시 디자인팀 협의

**결과:** 핸드오프 품질 향상, 재작업 감소

## 사례 3: 기능 범위 논쟁

**상황:** 클라이언트 추가 요청 vs 팀 부담

**PM 박서연:**
> "클라이언트가 3D 뷰어 기능을 요청했습니다."

**CTO 이준혁:**
> "현재 일정 내 고품질 구현은 어렵습니다."

**해결 과정:**
1. 3D 뷰어의 비즈니스 가치 분석
2. 간소화 버전 MVP 제안
3. 클라이언트와 협상
4. 2D 이미지 360도 뷰어로 타협

**결과:** 비용 효율적 대안 도출, 클라이언트 만족

# 지식 공유 활동

## 1. 기술 세미나

- **주기:** 격주 금요일 오후 5시
- **형식:** 팀원 발표 (30분) + Q&A (15분)
- **주제 예시:**
  - Next.js 14 App Router 베스트 프랙티스 (오승우)
  - PostgreSQL JSONB 성능 최적화 (황정민)
  - Figma 디자인 시스템 구축기 (강민수)
  - Playwright E2E 테스트 자동화 (김서윤)

## 2. 코드 리뷰 문화

- 모든 PR은 최소 1명 이상 리뷰 필수
- 시니어 개발자는 주니어 코드 멘토링
- 주간 '코드 리뷰 하이라이트' 공유

## 3. 문서화

- 모든 기술 결정 ADR(Architecture Decision Record) 기록
- API 문서 자동화 (Swagger)
- 컴포넌트 문서화 (Storybook)
- 온보딩 가이드 유지보수
